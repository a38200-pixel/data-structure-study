# 02. 복잡도 분석

## 1. 복잡도 분석이 필요한 이유

알고리즘을 비교할 때 단순히 실행 시간을 직접 측정하는 것만으로는 부족합니다.  
컴퓨터 성능, 실행 환경, 데이터 상태에 따라 시간이 달라질 수 있기 때문입니다.

그래서 알고리즘은 보통 입력 크기 `n`이 증가할 때 연산 횟수가 어떻게 증가하는지를 기준으로 분석합니다.

## 2. 시간 복잡도와 공간 복잡도

| 구분 | 의미 |
|---|---|
| 시간 복잡도 | 입력 크기가 증가할 때 실행 시간이 증가하는 정도 |
| 공간 복잡도 | 입력 크기가 증가할 때 필요한 메모리가 증가하는 정도 |

자료구조 학습에서는 보통 시간 복잡도를 먼저 중요하게 봅니다.

## 3. 점근적 표기법

점근적 표기법은 입력 크기 `n`이 충분히 커졌을 때 알고리즘의 증가 속도를 단순화해서 표현하는 방법입니다.

예를 들어 다음 함수가 있다고 하면,

```text
T(n) = n² + n + 1
```

입력이 커질수록 `n²` 항의 영향이 가장 커지므로 보통 `O(n²)`로 표현합니다.

## 4. Big-O 표기법

Big-O는 함수의 **상한**을 나타냅니다.  
알고리즘이 최악의 경우 어느 정도까지 느려질 수 있는지를 표현할 때 자주 사용합니다.

| 표기 | 의미 | 예시 |
|---|---|---|
| O(1) | 상수 시간 | 배열 인덱스 접근 |
| O(log n) | 로그 시간 | 이진 탐색 |
| O(n) | 선형 시간 | 순차 탐색 |
| O(n log n) | 선형 로그 시간 | 병합 정렬, 평균 퀵 정렬 |
| O(n²) | 이차 시간 | 선택 정렬, 삽입 정렬, 버블 정렬 |
| O(2ⁿ) | 지수 시간 | 일부 완전 탐색 |
| O(n!) | 팩토리얼 시간 | 순열 기반 완전 탐색 |

## 5. Big-Ω와 Big-Θ

| 표기 | 의미 |
|---|---|
| Big-O | 상한 |
| Big-Ω | 하한 |
| Big-Θ | 상한과 하한이 같은 경우 |

일반적인 학습과 코딩테스트에서는 Big-O 표기를 가장 많이 사용합니다.

## 6. 최선, 평균, 최악의 경우

입력 데이터의 상태에 따라 같은 알고리즘도 실행 시간이 달라질 수 있습니다.

| 경우 | 의미 |
|---|---|
| Best Case | 가장 빠르게 끝나는 경우 |
| Average Case | 평균적인 입력에서의 경우 |
| Worst Case | 가장 오래 걸리는 경우 |

예를 들어 순차 탐색은 첫 번째 원소가 정답이면 `O(1)`이지만, 마지막 원소가 정답이거나 데이터가 없으면 `O(n)`입니다.

## 7. 예시: 순차 탐색과 이진 탐색

```python
def sequential_search(arr, key):
    for i, value in enumerate(arr):
        if value == key:
            return i
    return -1
```

순차 탐색은 처음부터 끝까지 하나씩 확인하므로 시간 복잡도는 `O(n)`입니다.

```python
def binary_search(arr, key):
    low, high = 0, len(arr) - 1

    while low <= high:
        mid = (low + high) // 2

        if arr[mid] == key:
            return mid
        elif arr[mid] < key:
            low = mid + 1
        else:
            high = mid - 1

    return -1
```

이진 탐색은 매 단계마다 탐색 범위를 절반으로 줄이므로 `O(log n)`입니다.  
단, **정렬된 데이터**에서만 사용할 수 있습니다.

## 8. 핵심 정리

- 복잡도 분석은 입력 크기 증가에 따른 효율을 분석하는 방법이다.
- Big-O는 알고리즘의 상한을 표현한다.
- `O(1)`, `O(log n)`, `O(n)`, `O(n log n)`, `O(n²)` 순서로 일반적으로 더 느려진다.
- 이진 탐색은 빠르지만 정렬된 데이터가 필요하다.
- 알고리즘 선택 시 최악의 경우 성능을 고려해야 한다.
